# AgentOS v0.1.0 — Feature Design Document

> 核心执行链路可用，Member 可以在项目中完成任务。

---

## Part A — 版本级叙事

### A.1 版本目标

v0.1.0 的核心目标是：**打通 Member 从登录到完成任务的完整执行链路**。

具体包括：
1. Member 登录后看到个人数据看板 + HubAgent 对话区
2. Member 可以创建项目，或进入已有项目
3. 所有项目成员均可在项目内创建自己的任务
4. 成员在任务页与 WorkAgent 协作，Sub-Agent 执行具体操作
5. Sub-Agent 触发审批时，任务创建者在自己的 HubAgent 处理，执行自动恢复

### A.2 本版本范围

**包含（In Scope）**：
- 全局布局与导航（Sidebar + 两栏布局）
- 账号登录与类型识别（Member/Admin）
- Member 首页（数据看板 + HubAgent 对话区）
- 项目创建与管理
- 任务创建与执行
- WorkAgent 对话与项目上下文记忆
- Sub-Agent 执行流与审批中断恢复
- HubAgent 审批汇集与处理
- 任务页聊天输入与附件引用（F-033）
- Sidebar 项目置顶/重命名/归档（F-034）
- 项目页文件 Tab（上下文 + 产物）（F-035）
- 任务页项目树（上下文 + 产物）（F-036）

**不包含（Out of Scope）**：
- Admin 组织看板（F-008, F-009）— P1，v0.2.0
- 独立任务页（F-005）— P1，v0.2.0
- 任务树实时面板（F-019）— P1，v0.2.0
- Skills/Tools 管理（F-022 ~ F-026）— P1，v0.2.0
- 站内通知中心（F-028）— P2，v0.3.0
- 知识图谱（F-029, F-030）— P2，v0.3.0
- 项目归档（F-032）— P3，v0.4.0

### A.3 MVP 主流程

```
[登录]
  ↓
[Member 首页] ──→ 数据看板（个人维度）+ HubAgent 对话区
  ↓                                    ↓
[创建项目] ◄───────────────────────────┘（HubAgent 对话触发）
  ↓
[项目页] ──→ 任务列表 + WorkAgent 对话区
  ↓
[创建任务]（所有成员均可）
  ↓
[任务页] ──→ 执行流 + WorkAgent + Sub-Agent
  ↓
[Sub-Agent 触发审批]
  ↓
[HubAgent 审批卡片] ──→ 任务创建者处理
  ↓
[Sub-Agent 恢复/终止]
  ↓
[任务完成] ──→ 产出写入个人项目记忆
```

### A.4 验收标准

**系统级**：
- [ ] Member 可以从登录到完成任务，全程无需离开平台
- [ ] 所有 P0 Feature 状态为 Completed
- [ ] 无 High Priority 已知问题

**体验级**：
- [ ] 页面加载 < 1.5s
- [ ] WebSocket 消息延迟 < 200ms
- [ ] 移动端不崩溃（最小支持 1280px，但缩放不崩）

**数据级**：
- [ ] 项目创建后该用户的 WorkAgent 实例同步初始化
- [ ] 任务完成后该用户的个人项目记忆自动更新
- [ ] 审批结果实时同步到任务页

### A.5 前端基线

- React 19 + TypeScript + Vite
- Tailwind CSS v4
- Lucide React 线稿图标
- Canvas 2D（力导向图谱保留，但本版本不强制要求完全可用）
- 所有 Mock 数据硬编码在 `src/data/mockData.ts`

### A.6 架构基线

- 前端 Mock 数据，无后端 API
- WebSocket 使用 mock（模拟实时效果）
- 状态管理：React Context + useState/useReducer
- 路由：React Router（或类似方案）

## FEATURE_F-001: 全局两栏布局与 Sidebar

### 1. 需求概述

所有页面采用统一的两栏布局：左侧 Sidebar（240px，可折叠为 56px）+ 右侧主内容区。Agent 对话区嵌入主内容区右列，不再有全局常驻第三栏。

### 2. 影响范围

- **页面**：所有页面（首页、Skills、Tools、项目页、任务页）
- **组件**：Sidebar、Layout、App.tsx
- **样式**：全局布局 CSS、Tailwind 配置

### 3. 技术方案

```
Layout Component
├── Sidebar（固定左侧，240px / 56px）
│   ├── Logo
│   ├── 导航组（首页/Skills/Tools）
│   ├── 新建对话
│   ├── 项目树
│   └── 用户卡片
└── Main Content（flex-1，弹性填充）
    └── 页面级组件
```

- Sidebar 状态：`isCollapsed`（localStorage 持久化）
- 折叠动画：CSS transition，200ms ease
- Tooltip：折叠时 hover 显示导航名称

### 4. 接口契约

```typescript
interface SidebarProps {
  userType: 'member' | 'admin';
  projects: Project[];
  activeRoute: string;
  isCollapsed: boolean;
  onToggleCollapse: () => void;
}

interface LayoutProps {
  children: React.ReactNode;
  rightPanel?: React.ReactNode; // Agent 对话区
}
```

### 5. 实现步骤

1. 创建 `Layout` 组件，实现两栏结构
2. 创建 `Sidebar` 组件，实现展开/折叠
3. 实现导航激活状态
4. 实现项目树渲染（ProjectTree 组件）
5. 实现用户卡片（UserCard 组件）
6. 在 App.tsx 中包裹所有路由

### 6. 验收标准

- [ ] 两栏布局在所有页面一致
- [ ] Member / Admin 渲染不同的 Sidebar
- [ ] 折叠为 56px 图标模式，hover 显示 Tooltip
- [ ] 项目有活跃任务时显示 `●` 标识
- [ ] 「新建对话」区域可折叠
- [ ] 路由切换时导航激活状态正确
- [ ] 折叠状态刷新后保持

## FEATURE_F-002: 账号登录与类型识别

### 1. 需求概述

用户通过企业邮箱登录，系统根据账号类型（Member / Admin）渲染不同的首页和权限体系。账号类型由 IT 管理员在管理后台配置，登录后固定，不可自行切换。

### 2. 影响范围

- **页面**：登录页、首页、所有需要权限判断的页面
- **组件**：AuthContext、ProtectedRoute
- **数据**：User 类型、权限判断逻辑

### 3. 技术方案

- Mock 登录：固定账号（member / admin），输入任意密码即可登录
- 用户类型存储：localStorage `agentos_user`
- 权限守卫：ProtectedRoute 组件，Member 访问 Admin 路由 → 302 重定向

```
App.tsx
├── AuthProvider（Context）
│   ├── user: User | null
│   ├── login(email, password)
│   ├── logout()
│   └── isAdmin()
├── ProtectedRoute
│   └── 检查 userType，不匹配则重定向
└── Routes
    ├── /dashboard → DashboardPage（根据 userType 渲染不同内容）
    ├── /skills → SkillsPage
    ├── /tools → ToolsPage
    ├── /project/:id → ProjectPage
    ├── /settings/* → AdminOnly（ProtectedRoute）
    └── /org/* → AdminOnly（ProtectedRoute）
```

### 4. 接口契约

```typescript
interface User {
  id: string;
  email: string;
  name: string;
  avatar: string;
  type: 'member' | 'admin';
}

interface AuthContextType {
  user: User | null;
  isLoading: boolean;
  login: (email: string, password: string) => Promise<void>;
  logout: () => void;
  isAdmin: () => boolean;
}
```

### 5. 实现步骤

1. 创建 `AuthContext` 和 `AuthProvider`
2. 创建 `ProtectedRoute` 组件
3. 创建 Login 页面（企业邮箱输入 + 密码）
4. 在 App.tsx 中配置路由守卫
5. 实现 Mock 登录逻辑（固定两个账号）

### 6. 验收标准

- [ ] 登录成功后根据账号类型跳转不同首页
- [ ] Member 访问 Admin 路由被重定向到首页
- [ ] 首次登录时 HubAgent 完成初始化（Mock）
- [ ] 刷新页面后登录状态保持
- [ ] 登出后清除所有状态

## FEATURE_F-004: HubAgent 对话区

### 1. 需求概述

HubAgent 对话区是 Member 与 HubAgent 交互的统一界面，在首页、Skills 页、Tools 页的右列复用同一组件。页面切换时对话历史保留，上下文随场景自动切换。

### 2. 影响范围

- **页面**：首页、Skills 页、Tools 页
- **组件**：HubAgentChat、ChatMessage、ChatInput
- **状态**：对话历史、上下文模式

### 3. 技术方案

```
HubAgentChat（可复用组件）
├── Header
│   ├── Agent 名称 + 状态指示器
│   └── 上下文切换提示（2s 消失）
├── MessageList
│   ├── UserMessage（灰色气泡）
│   ├── AgentMessage（无气泡，纯文本）
│   └── SystemMessage（提示消息）
└── InputArea
    ├── Textarea（Shift+Enter 换行）
    └── SendButton
```

- 对话历史存储：localStorage（按用户隔离）
- 上下文模式：`'global' | 'skill' | 'tools'`，由页面传入
- 流式输出：模拟打字机效果（Mock）

### 4. 接口契约

```typescript
interface HubAgentChatProps {
  contextMode: 'global' | 'skill' | 'tools';
  className?: string;
}

interface ChatMessage {
  id: string;
  sender: 'user' | 'hubagent';
  content: string;
  timestamp: string;
}
```

### 5. 实现步骤

1. 创建 `HubAgentChat` 组件框架
2. 创建 `ChatMessage` 子组件（区分用户/Agent 样式）
3. 创建 `ChatInput` 组件（Enter 发送，Shift+Enter 换行）
4. 实现对话历史存储/读取（localStorage）
5. 实现上下文切换提示动画
6. 在首页/Skills/Tools 页复用

### 6. 验收标准

- [ ] 流式输出正常渲染，支持 Markdown
- [ ] 页面切换时对话历史不丢失
- [ ] 上下文切换提示短暂显示后消失
- [ ] `Enter` 发送，`Shift+Enter` 换行
- [ ] 空状态展示快捷引导问题
- [ ] 对话历史按用户隔离

## FEATURE_F-006: HubAgent — 审批汇集与处理

### 1. 需求概述

用户自己所有项目 WorkAgent 上报的审批请求，统一汇集在该用户自己的 HubAgent 对话区以审批卡片形式展示。这是用户处理自己任务中 Sub-Agent 审批的唯一入口。

### 2. 影响范围

- **页面**：首页（HubAgent 对话区）
- **组件**：ApprovalCard、HubAgentChat（扩展）
- **状态**：审批列表、审批处理流程

### 3. 技术方案

```
HubAgentChat
├── ...（现有消息列表）
└── ApprovalCard（特殊消息类型）
    ├── 来源信息（项目/任务）
    ├── 操作详情（API / 数量 / 费用）
    ├── WorkAgent 建议
    └── 操作按钮（查看详情 / 拒绝 / 批准）
```

- 审批数据存储：Mock 数组，模拟实时推送
- 处理结果回传：通过 Context 通知 WorkAgent（Mock）
- 每个用户仅在自己的 HubAgent 中收到自己任务的审批请求

### 4. 接口契约

```typescript
interface ApprovalRequest {
  id: string;
  projectId: string;
  projectName: string;
  taskId: string;
  taskName: string;
  action: string;
  estimatedCost: number;
  workAgentSuggestion: string;
  status: 'pending' | 'approved' | 'rejected';
  createdAt: string;
}

interface ApprovalCardProps {
  approval: ApprovalRequest;
  onApprove: (id: string) => void;
  onReject: (id: string, reason?: string) => void;
}
```

### 5. 实现步骤

1. 创建 `ApprovalCard` 组件
2. 扩展 HubAgentChat 支持审批卡片消息类型
3. 实现审批处理逻辑（批准/拒绝）
4. 实现审批结果回传（Mock WebSocket）

### 6. 验收标准

- [ ] 审批卡片在 WorkAgent 上报后实时出现
- [ ] 批准后 Sub-Agent 自动恢复执行
- [ ] 拒绝时支持填写原因，随结果回传
- [ ] 处理后卡片状态更新，KPI 卡片数字同步减少
- [ ] 用户不会收到他人任务的审批卡片

## FEATURE_F-014: WorkAgent — 项目上下文记忆

### 2. 影响范围

User Project Memory（Mock 数据结构，每个用户独立）
├── taskSummaries: TaskSummary[]      // 该用户创建的任务产出摘要
├── knowledgeGraph: KnowledgeGraph    // 该用户个人知识图谱
├── projectGoals: string[]            // 该用户设定的项目目标
├── constraints: string[]             // 该用户设定的约束
└── userProfile: UserProfile          // 该用户在此项目中的画像

上下文引用触发
├── 用户输入任务指令
├── WorkAgent 检索相关历史摘要
├── 自动引用并展示
└── 用户确认或修改方案

## FEATURE_F-018: 任务页 — Sub-Agent 审批中断与恢复

### 1. 需求概述

Sub-Agent 触发审批策略时，执行流中插入审批中断卡片，任务暂停。审批请求发送至任务创建者自己的 HubAgent，由其处理后 Sub-Agent 自动恢复或终止。

### 2. 影响范围

- **页面**：任务页、首页（HubAgent）
- **组件**：ApprovalInterruptCard、HubAgent（扩展）
- **状态**：审批状态、任务暂停/恢复

### 3. 技术方案

```
审批触发
├── Sub-Agent 执行中触发审批策略
├── WorkAgent 暂停 Sub-Agent
├── 执行流中插入 ApprovalInterruptCard
├── 推送审批请求至任务创建者自己的 HubAgent
└── 任务页显示「等待审批」状态

审批处理
├── 任务创建者在自己的 HubAgent 处理（批准/拒绝）
├── 结果回传至 WorkAgent
├── 任务页卡片状态更新
└── Sub-Agent 恢复执行或终止

任务页操作
├── [去 HubAgent 处理] → 跳转到首页 HubAgent 对话区
└── [跳过此步骤] → 继续执行（仅任务创建者）
```

### 4. 接口契约

```typescript
interface ApprovalInterrupt {
  id: string;
  subAgentName: string;
  action: string;
  estimatedCost: number;
  taskCreatorId: string;  // 审批发送给任务创建者
  status: 'pending' | 'approved' | 'rejected' | 'skipped';
  requestedAt: string;
  resolvedAt?: string;
  resolution?: 'approved' | 'rejected' | 'skipped';
  rejectReason?: string;
}

interface ApprovalInterruptCardProps {
  interrupt: ApprovalInterrupt;
  onUrge: () => void;
  onSkip: () => void;
  canUrge: boolean;
  canSkip: boolean;
  cooldownRemaining?: number;
}
```

### 5. 实现步骤

1. 创建 `ApprovalInterruptCard` 组件
2. 扩展执行流支持审批中断消息类型
3. 实现催审批逻辑（1h 冷却）
4. 实现跳过步骤逻辑
5. 实现审批结果实时更新（Mock WebSocket）
6. 与 HubAgent 审批卡片联动

### 6. 验收标准

- [ ] 审批中断卡片在正确时机插入执行流
- [ ] 审批请求正确发送至任务创建者的 HubAgent
- [ ] 跳过后任务继续，完成汇总有标注
- [ ] 审批结果通过 WebSocket 实时更新卡片状态
- [ ] 批准后 Sub-Agent 自动恢复，拒绝后终止分支
