AgentOS Requirement Brief · v0.3

AgentOS 是 KodaX 生态中更懂企业的管理智能体操作层。

本文档基于当前产品矩阵与近期讨论整理，目标是明确 AgentOS 的产品定位、三类核心 Agent 的职责边界、Agent 之间的数据与任务通讯机制，以及从 AgentOS 到 Space 的企业级执行闭环。

KodaX CLI：更懂企业的软件设施与流程 KodaX Space：更懂企业员工的本地执行端 AgentOS：更懂企业的管理智能体层 统一内核：KodaX Agent Kernel

一、背景与产品矩阵

KodaX 生态不是三款互相竞争的产品，而是三个层级清晰的企业 Agent 产品形态。

KodaX CLI

软件设施与流程能力

面向研发、运维、软件流程和代码工程。它是更底层的 Coding / DevOps / 软件设施能力入口，也是 KodaX Agent 能力的基础表达。

KodaX Space

员工本地执行端

面向个人电脑、文件、浏览器、账号、本地工具和端侧权限。它更懂企业员工，承接真实执行任务。

AgentOS

企业管理智能体层

面向企业、组织、项目、记忆、审批、Skill、Connector 和跨成员协同。它不直接做本地任务，而是管理和编排企业级上下文。

产品 核心定位 核心对象 边界
KodaX CLI 企业软件设施、代码、流程与运维能力 代码库、命令行、软件工程流程、DevOps 不承载企业项目协同和组织级治理
KodaX Space 企业员工本地执行环境 本地文件、浏览器、IDE、账号、桌面任务 不负责组织记忆、项目总控和企业市场治理
AgentOS 企业级 Agent 管理操作系统 项目、员工、审批、企业记忆、Skill、Connector、PAT 不直接替代 Space 执行任务，负责上层管理和调度

二、AgentOS 的产品定位

AgentOS 的主线不是像 Space 一样执行任务，而是作为更上层的管理智能体系统。

一句话定义

AgentOS 是企业项目、组织记忆、工具市场、权限治理与本地 Space 执行之间的协同操作层。

管理

管理企业协同

管理项目、成员、审批、工作状态、企业效能和组织级上下文。

治理

治理企业资产

治理企业记忆库、技能市场、连接器市场、数据权限和上架审批。

通讯

连接 Space 执行

通过 PAT 把数据和任务请求送到 Space，再把执行结果回传给对应 Agent。

三、AgentOS 的三个核心 Agent

AgentOS 由 PAT Agent、OrgAgent、ProjectAgent 三个主要智能体构成。它们共享 KodaX 内核，靠角色配置、权限、工具和上下文区分职责。

Agent 定位 主要能力 不负责什么
PAT Agent / PersonalAgent AgentOS 与 Space 之间的数据与任务通讯层 数据请求转发、任务分发、权限确认、Space 任务创建、结果回传、消息提醒 不做组织治理，不做项目总控，不直接替代 Space 执行
OrgAgent 企业级治理与工具市场管理 Agent 首页数据看板、审批中心、企业记忆库、技能市场、连接器市场、管理视角治理 不负责具体项目任务拆解，不直接进入员工本地电脑执行
ProjectAgent 项目级管理与协同 Agent 创建项目、项目看板、负责人总控、成员协同、任务拆解、分发到 PAT、成果审核和项目记忆沉淀 不直接执行成员本地任务，不管理全局市场和组织级审批

四、PAT Agent 的通讯设计

PAT 是通讯兵。它负责在 AgentOS 与 Space 之间搬运结构化请求和结果，同时维护权限、来源、项目归属和用户确认。

数据通道

Data Channel

  • Space 下载 Skill，例如行业研究、报告生成、数据清洗 Skill。
  • Space 连接 Connector / MCP，例如企查查、企业搜索、CRM、GitHub。
  • Space 请求数据库或数据源访问，例如只读数据仓库、项目文件索引。
  • Space 回传结构化摘要、证据、哈希、字段口径和结果文件索引。
任务通道

Task Channel

  • 本人主动发起的任务请求。
  • ProjectAgent 分发给成员的项目任务。
  • 同项目其他成员产生的协作消息。
  • Space 完成后通过 PAT 回传到对应项目页和 ProjectAgent。

PAT 的核心模块

模块 能力 关键要求
消息收件箱 接收 ProjectAgent、OrgAgent、Space 的任务和数据消息 按通道分类，支持未读、提醒、回执和状态追踪
权限确认 对本地文件、账号、数据源、Connector、Skill 下载进行确认 敏感数据必须先确认范围，默认回传摘要和证据
Space 调用 在 Space 创建任务、安装 Skill、连接工具、启动执行 执行发生在 Space，AgentOS 只管理请求和回传结果
结果回传 把 Space 的执行结果送回 ProjectAgent 或 OrgAgent 回传必须携带项目、成员、来源、证据和权限状态

五、OrgAgent 需求设计

OrgAgent 管理企业级工具、技能、记忆、连接器和审批治理，是 AgentOS 的组织级管理中枢。

首页

个人视角与管理视角

  • 个人视角展示个人效能、待处理事项、Space 回传、个人项目贡献和 PAT 状态。
  • 管理视角展示员工数据、组织效能、Token 消耗、项目价值、成员负载和 Space 在线状态。
  • 管理视角提供统一审批中心，集中审批企业记忆、技能市场和连接器。
市场治理

企业级能力资产

  • 企业记忆库：上传文档和项目记忆，形成可视化、可问答、可提炼的组织记忆。
  • 技能市场：用户可创建 Skill，管理员审批后上架，Space 可下载调用。
  • 连接器：MCP、API、数据库、数据服务等企业级工具，审批后按权限开放。

OrgAgent 管理对象

对象 用户动作 管理员动作 Space 关系
企业记忆库 上传文档、提交项目记忆候选、发起问答和提炼 审核、脱敏、去重、归类、发布到全局记忆 Space 回传成果可沉淀为项目记忆，进一步进入企业记忆
技能市场 按需求创建 Skill、提交上架、维护版本 审批权限、测试样例、依赖工具和安全风险 审批通过后，Space 可下载并执行 Skill
连接器 申请接入 MCP、API、数据库和数据服务 审批访问范围、授权周期、敏感字段和审计规则 Space 可按权限连接工具和数据源

六、ProjectAgent 需求设计

ProjectAgent 是项目级管理 Agent。它把项目从“个人文件和零散任务”变成“可视化项目空间和跨成员协同对象”。

创建

任意用户可创建项目

用户创建项目后自动成为项目负责人，也就是项目群主。项目会出现在负责人和被拉入成员的 AgentOS 页面。

协同

项目空间不是群聊

项目页像项目看板，展示节点、成员、责任、进度、审批和成果。右侧 ProjectAgent 用于项目上下文问答与管理。

分发

任务通过 PAT 到 Space

负责人给成员分发任务后，任务进入成员 PAT，由 PAT 在成员 Space 创建并执行，完成后再回传项目页。

项目核心流程

创建项目负责人上传资料，并与 ProjectAgent 对齐项目目标。
拆解节点ProjectAgent 基于企业记忆拆解项目阶段和责任项。
拉入成员负责人拉起成员，形成项目协作空间和成员视图。
分发 PAT任务发送给成员 PAT，并在对应 Space 创建执行任务。
回传沉淀Space 结果经 PAT 回传，ProjectAgent 审核、统计和沉淀。
角色 看到什么 能做什么 不能做什么
项目负责人 完整项目图谱、成员分工、节点进度、回传结果、审批状态 与 ProjectAgent 对话、拆解任务、拉成员、分发任务、审核成果 不能直接进入成员本地 Space 执行或读取其本地文件
项目成员 自己参与的项目页、责任项、上下文、审批和回传结果 用自己的 ProjectAgent 问答、确认任务、通过 PAT 在 Space 执行 不能管理全局项目权限或替负责人审核全项目成果
ProjectAgent 项目上下文、企业记忆、成员状态、任务节点、成果证据 项目规划、问答、拆解、分发、审核、统计和项目记忆沉淀 不直接做本地执行，不替代 PAT 或 Space

七、Agent 间通讯协议建议

内部不建议完全依赖外部 A2A 协议，也不建议从零做封闭协议。推荐内部定义 AgentOS Message Protocol，同时保留 A2A 适配能力。

内部协议

AgentOS Message Protocol

用于 ProjectAgent、OrgAgent、PAT、Space 之间传递项目、数据、权限、任务和回传结果。

外部互操作

A2A Adapter

用于未来接入第三方 Agent 或把 AgentOS Agent 暴露给外部系统。它是适配层，不是内部全部语义的唯一来源。

工具调用

MCP / Connector

用于 Skill、MCP、数据库、API 和外部系统的能力调用，由 OrgAgent 管理，由 PAT 下发到 Space。

八、关键协同链路

下面两条链路是 AgentOS 是否成立的核心闭环。

链路 A：ProjectAgent 分发项目任务给成员

负责人 ProjectAgent拆解任务，指定成员、上下文和交付标准。
成员 PAT收到任务消息，提醒成员确认并准备发送到 Space。
成员 Space创建本地任务，调用 KodaX 内核、Skill 和本地工具执行。
成员 PAT接收 Space 结果，进行摘要、证据和权限检查。
项目页结果回到项目页，由负责人 ProjectAgent 审核与沉淀。

链路 B：OrgAgent 发起数据或市场能力请求

OrgAgent发起企业记忆、Skill、Connector 或数据相关请求。
本人 PAT判断数据通道，确认权限、范围和本地执行要求。
本人 Space下载 Skill、连接 Connector、访问数据源或执行分析。
本人 PAT回收结果，默认回传摘要、证据和状态。
OrgAgent更新审批、市场状态、企业记忆或组织数据看板。

十、产品边界与原则

这些边界能避免 AgentOS、Space、KodaX CLI 相互重叠。

应该做
  • 管理项目、组织、审批、企业记忆、技能市场和连接器。
  • 通过 PAT 把数据和任务请求送到 Space。
  • 沉淀项目上下文、执行证据和可复用经验。
  • 把 KodaX 内核作为统一 Agent 能力底座。
不应该做
  • 不在 AgentOS 内直接执行员工本地任务。
  • 不重复构建一套新的 Agent 内核。
  • 不把项目页做成普通群聊页面。
  • 不让 Skill、Connector、企业记忆绕过审批直接全局可用。
