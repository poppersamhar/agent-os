Enterprise Agent Operating Layer

AgentOS 是企业项目、组织记忆、权限治理与端侧执行之间的协同操作层。

这份文档聚焦 AgentOS 当前应呈现的产品形态：Agent 能力不再以每个页面右侧聊天框的方式堆叠，而是通过 PAT、项目上下文服务、企业记忆库、Skill 与 Connector 治理进入企业协作流程。

一、Agent 设计：一个内核，多种角色配置

KodaX 是唯一 Agent 内核。AgentOS 中不同 Agent 的差异来自角色配置、Skill、工具、上下文和权限，不来自重写能力。

Role Config, Not New Kernel
角色 产品形态 核心职责 是否执行任务
PersonalAgent / PAT 驻扎在 Space 与 AgentOS 用户区的小型通讯入口 请求提醒、审批提醒、数据授权、任务包转发、结果回传 不直接执行；执行由 Space 内 KodaX 完成
OrgAgent 组织治理服务，可在企业记忆库、Skill、连接器页面出现 企业记忆治理、Skill 管理、Connector 权限、组织级规则与审核 不做员工任务；做治理、审核、归档和配置
Project Context Assistant 项目页上下文服务，不再强调“ProjectAgent 人设” 项目问答、项目计划、节点解释、证据追踪、项目记忆提炼 不做具体工作；只组织项目上下文
Professional Agent / Skill Agent 通过 Skill 市场或配置注册 专业能力封装，例如市场分析、PPT、数据清洗、合规检查 可被 Space 调用执行，也可被 AgentOS 编排为能力资源

二、页面形态建议

页面是工作对象的容器，不是 Agent 的展示位。Agent 能力隐藏在按钮、流程、提醒和上下文服务后面。

UI 信息架构
页面 主要对象 建议展示 不建议展示
首页 个人请求与项目状态 请求中心、活跃项目、Space/PAT 状态、记忆候选、待审批 右侧 PersonalAgent 聊天面板
项目页 项目上下文 项目图谱、节点、负责人、证据、决策、项目记忆、项目问答入口 执行型 ProjectAgent、任务创建层、员工本地执行能力
企业记忆库 组织资产 全局记忆、项目记忆候选、脱敏审核、权限、引用关系图谱 普通文件上传库的表达
Skills 可复用能力 Skill 市场、创建/配置/权限、成功率、调用日志、版本治理 把 Skill 做成独立聊天机器人列表
连接器 系统接入 MCP/API/第三方系统/数字员工接入、权限范围、审计日志 无权限边界的全局工具箱

三、关键流程：项目、执行、记忆三条闭环

AgentOS 的产品价值来自闭环，而不是单个问答能力。

Operating Loop

1. 项目协同闭环

项目进入负责人创建项目文档和目标约束
企业知识确认基于企业记忆库确认方向和历史经验
节点拆解拆为项目节点、负责人和交付证据
PAT 下发发送到成员 Space，由员工授权执行
结果回传证据、摘要、决策进入项目页

2. 端侧执行闭环

AgentOS 请求项目请求、审批请求、数据请求结构化
PAT 提醒小猫在用户区提醒并解释请求
Space 授权员工决定是否允许读取、执行、回传
KodaX 执行调用内核、工具、Skill 完成具体任务
AgentOS 沉淀只回传必要摘要、证据和结构化结果

3. 企业记忆闭环

项目记忆沉淀任务、决策、证据和复盘
提炼候选项目上下文服务生成可复用经验
脱敏审核负责人确认权限、脱敏和适用场景
企业记忆OrgAgent 归类、打标、发布
反哺项目新项目可引用全局经验和规则

六、产品优缺点

AgentOS 的价值很清晰，但难点也不小：它要同时解释清楚企业协同、端侧执行、记忆治理和 Agent 内核复用。

Strengths & Risks

优势 1：单一 Agent 内核，避免能力分裂

KodaX 作为唯一内核，AgentOS 不需要重复构建 Agent 执行能力，产品矩阵更稳。

优势 2：项目经验可沉淀为组织资产

项目页不只是协作空间，还能把决策、证据、复盘提炼成企业记忆。

优势 3：端侧隐私和权限边界更自然

Space/PAT 处理本地文件和账号授权，AgentOS 只接收必要摘要和证据。

优势 4：适合企业治理场景

审批、脱敏、Skill 权限、Connector 审计天然属于 AgentOS，而不是个人电脑端。

风险 1：概念层次容易过多

KodaX、Space、PAT、AgentOS、Project Context、OrgAgent 都要解释清楚，否则用户会困惑。

风险 2：必须依赖 Space 成熟度

如果 Space 的端侧执行、权限弹窗、任务回传不稳定，AgentOS 的闭环会断。

风险 3：企业系统集成成本高

连接器、权限、审计、数据口径和组织架构同步，会决定产品能否进入真实业务。

风险 4：需要避免“又一个项目管理工具”

AgentOS 应定位为 AI-native 项目上下文与记忆层，而不是替代 Jira、飞书项目、钉钉项目。

七、落地路线建议

先把边界和闭环做准，再扩展组织级生态。

Next Steps

阶段 1：产品形态收敛

首页去 PersonalAgent 面板；项目页去执行型 ProjectAgent；PAT 小猫成为个人请求入口；项目页聚焦图谱和上下文。

阶段 2：打通请求闭环

项目请求、审批请求、数据请求从 AgentOS 到 PAT，再到 Space 执行，最后回传摘要、证据和状态。

阶段 3：建设企业记忆

项目记忆候选、脱敏审核、负责人确认、OrgAgent 归类发布，形成组织级复用资产。

阶段 4：Skill 与 Connector 治理

建立 Skill 市场、版本、权限、调用日志；Connector 支持企业系统和第三方数字员工接入。

阶段 5：与外部协同平台集成

接入飞书、钉钉、Jira、GitHub、Slack、SharePoint 等，而不是试图替代所有入口。

阶段 6：形成行业方案

围绕供应链、产品研发、售前、数据治理、运营分析等场景，封装可复制的项目模板和记忆模型。
